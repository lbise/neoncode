# From validated POC to product foundation

## Status

The Windows Terminal embedding POC succeeded. The validated snapshot is tagged:

```text
poc/windows-terminal-embedded
```

The POC proved the core path:

```text
Windows WPF app
  ⇄ embedded Windows Terminal renderer
  ⇄ WebSocket protocol
  ⇄ Rust neoncode-hub
  ⇄ Linux/WSL PTY
  ⇄ bash / tmux / Neovim / agents
```

This means the project should move from spike mode to a deliberate prototype/product foundation. We should not discard the POC knowledge, but we should clean up the implementation boundaries and tooling before adding larger features.

## What the POC taught us

### Architecture validated

- Rust is a good fit for the session hub.
- A simple WebSocket protocol is sufficient for terminal data, lifecycle, and resize events.
- Windows can host a real terminal renderer while the PTY lives in WSL/Linux.
- Neovim rendering works through the full stack.
- Publishing the frontend to a Windows-local folder is viable and avoids running the GUI from UNC/WSL paths.

### Windows Terminal embedding lessons

- `Microsoft.Terminal.Control.dll` is viable for rendering.
- The upstream WPF wrapper is small enough to vendor for now.
- Windows Terminal application configuration is a higher layer than the renderer. The embedded control will not automatically read Windows Terminal `settings.json`.
- We can still reuse user settings by parsing Windows Terminal's `settings.json` and mapping renderer-relevant values ourselves.
- The current dependency setup uses an older pinned Windows Terminal tag and local compatibility patches. This is acceptable for the POC, but product work needs a reproducible dependency workflow.

### GUI toolkit lessons

- WPF works for the Windows-native terminal host.
- Changing GUI stacks now would require revalidating native terminal hosting.
- WPF should remain the near-term Windows prototype shell until there is a concrete reason to move.
- Electron, Qt, WinUI 3, Avalonia, or other options can be reassessed later with the terminal/session model already proven.

## Product direction

The target product is a workspace/session cockpit, not an IDE clone. It should orchestrate existing durable tools:

- SSH
- WSL
- tmux
- Neovim
- pi/opencode or other terminal agents
- browsers
- X2Go or other remote desktop/session tools

The GUI should own overview, layout, launch orchestration, reconnection, and presentation. Durable working state should remain in existing tools and machines where practical.

## Recommended route

### Phase 0: preserve the spike

Done:

- Commit the POC.
- Tag the validated commit as `poc/windows-terminal-embedded`.
- Keep the POC code as a reference for future refactors.

### Phase 1: clean prototype foundation

Goal: turn the spike into a maintainable prototype without changing the validated architecture.

Work items:

- Keep the Rust hub, but reorganize it around explicit session management.
- Define protocol message types cleanly in Rust and document them.
- Add structured tracing/logging to the hub.
- Add a Windows publish script that builds/publishes the frontend and verifies native terminal files are present.
- Add a dependency bootstrap script for Windows Terminal source, vcpkg, NuGet restore, patch application, and native control build.
- Move local compatibility changes for Windows Terminal into explicit patch files or deterministic script steps.
- Add a first app config file for local settings.
- Add app-level error reporting for backend disconnected, native terminal failed to load, session exited, and protocol errors.

Avoid during this phase:

- major UI polish;
- changing GUI toolkit;
- adding many workspace features before session handling is solid.

### Phase 2: terminal correctness and usability

Goal: make the terminal experience trustworthy.

Work items:

- Confirm resize propagation with `stty size`.
- Verify Ctrl+C, Ctrl+D, Ctrl+Z, Alt combinations, arrows, function keys, Home/End, Page Up/Down.
- Verify mouse mode in Neovim and tmux.
- Implement copy/paste behavior.
- Support multiple simultaneous terminal controls.
- Add clean session kill/restart behavior.
- Add scrollback expectations and known limitations.
- Load font and theme settings.
- Optionally parse Windows Terminal `settings.json` for:
  - font face;
  - font size;
  - color scheme;
  - foreground/background;
  - cursor style;
  - selection color.

Validation commands:

```bash
ls --color=always
printf '\e[31mred\e[0m\n'
nvim
tmux
stty size
```

### Phase 3: session model

Goal: evolve from one hardcoded shell session to real session orchestration.

Work items:

- Multiple sessions in the hub.
- Session IDs generated by the backend or agreed by protocol.
- List active sessions.
- Attach/detach behavior.
- Reconnect from frontend without killing the PTY.
- Session exit status and reason reporting.
- Configurable launch profiles:
  - local WSL shell;
  - project directory shell;
  - SSH command;
  - tmux attach/create;
  - custom command.
- Basic persistence of recent sessions/workspaces.

### Phase 4: workspace cockpit features

Goal: build the actual value proposition beyond terminal embedding.

Work items:

- Workspace definitions.
- Tabs/panes managed by the GUI.
- Saved layouts.
- Project/machine overview.
- Quick launch commands.
- SSH/tmux profile integration.
- Status indicators for sessions and hosts.
- Browser launch/profile hooks.
- X2Go or external remote desktop launch hooks.
- Keyboard command palette or equivalent launcher.

Design rule:

- tmux may provide remote persistence, but the GUI should own the visible layout.

### Phase 5: dependency/toolchain hardening

Goal: make the Windows Terminal dependency sustainable.

Work items:

- Re-evaluate Windows Terminal version.
- Try a newer stable tag after the build process is scripted.
- Keep the dependency pinned.
- Store local patches in the repo.
- Add script validation that required Visual Studio components are installed.
- Prefer reproducible build paths and avoid UNC paths for Windows-native builds.
- Decide whether vendoring the WPF wrapper remains acceptable or whether to maintain a small first-party wrapper around `HwndTerminal`.

### Phase 6: GUI toolkit reassessment

Only reassess the GUI stack after the terminal/session model is real.

Evaluation questions:

- Can the candidate toolkit host the native terminal control reliably?
- Does it preserve keyboard input fidelity?
- Does it support good docking/tabs/layout?
- Does it work in the locked-down work environment?
- Does it reduce or increase dependency risk?
- Does it support the eventual cross-platform story?

Candidate stacks:

- Continue WPF for Windows.
- WinUI 3 native Windows shell.
- Qt/QML.
- Avalonia.
- Electron/web UI with either native control hosting or xterm.js.
- Separate native frontends per platform sharing the Rust hub protocol.

Current recommendation:

- Continue with WPF for the Windows prototype because it is already proven.
- Do not switch stacks until there is a working session/workspace model to compare against.

## Definition of the next milestone

The next milestone should be a usable prototype, not a polished product:

- launch app from published Windows folder;
- connect to local WSL hub;
- open multiple terminal tabs;
- run Neovim/tmux reliably;
- use configured font/theme;
- close/restart sessions cleanly;
- preserve enough session/workspace state to be useful during daily experimentation.
